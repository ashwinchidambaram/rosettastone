## Context management protocol (read this carefully — it governs the whole run)

### Worklog discipline

The worklog is your external memory. Update it after every meaningful action, not just at phase boundaries. A "meaningful action" is: a finding recorded, a decision made, a cross-cutting insight spotted, a NEEDS_HUMAN_REVIEW item logged, a pass section written to the report.

Worklog entries are short and structured. Example:
```markdown
### 2026-04-06 14:23 — Pass 1 — Found server/ vs server_stitch/ duplication
- server_stitch/ appears to be a parallel implementation from an earlier design iteration
- References: grep shows server_stitch is not imported anywhere in src/rosettastone/
- Recommendation: SAFE_DELETE after human verification
- Logged to report under Pass 1 cleanup inventory
- Commit: a1b2c3d
```

Prepend newest entries at the top of the Phase Log so the most recent context is what you see first after a compact.

### Commit discipline

Every commit message embeds a worklog-style summary so `git log --oneline` tells the story even if the worklog file is corrupted. Format:
```
<type>(<scope>): <one-line summary>

<2-4 line body describing what was added to the report and why>

Phase: <Pass N - name>
Worklog: docs/reviews/2026-04-06-worklog.md
```

Commit granularity: one commit per pass section is the floor, one commit per meaningful finding is the ceiling. If you write 10 distinct findings into Pass 1, that can be 1–3 commits. Don't go overboard, but also don't save everything for the end.

### Phase-boundary compaction

After completing each Pass (0 through 5), do this exact sequence:

1. Ensure everything for that pass is committed (`git status` shows clean)
2. Write a Phase Summary entry to the worklog under a new `## Phase N Summary` heading. This is a dense 5–10 bullet recap of what the pass found, what was deferred, what the next pass should know.
3. Commit the worklog update: `git commit -am "docs(worklog): pass N summary"`
4. Run `/compact` to compress conversation history
5. **Post-compact re-orientation protocol** (do this as your very first action after compact):
   - Read `docs/reviews/.active-prompt-product.md`
   - Read `docs/reviews/2026-04-06-worklog.md` (especially the Current Status section and the most recent Phase Summary)
   - Run `git log --oneline -20`
   - Run `git status` to confirm clean tree
   - Update the Current Status section of the worklog: "Resumed after compact at [timestamp], entering Pass N+1"
   - Only then start the next pass

Do not compact mid-pass. If you hit context pressure inside a pass, commit whatever partial progress you have with a clear `WIP` marker in the message, write a partial-pass note to the worklog, then compact and resume. The partial commit is your safety net.

### Drift detection

At the start of each pass, after re-orientation, do a quick sanity check:

- Are you on branch `review/product-2026-04-06`? If not, stop.
- Does the worklog's last entry match what `git log` shows as the most recent commit? If not, investigate the mismatch before proceeding.
- Does the target user definition in the worklog still match your mental model? If you feel yourself drifting to a different user, stop and re-read the definition.
- Are you still in plan-only mode? Any instinct to "just fix this one thing" while reviewing is drift — log it to the action list instead.

Drift caught early is cheap. Drift caught after three passes is a rewrite.

## Pass 1 — Bloat & Cleanup Inventory

Identify everything bloating the project without contributing value. This is an inventory, not an execution — every item goes in the report with a recommended action, but nothing gets deleted or moved.

Categories to hunt for:

- Tracked artifacts that belong in `.gitignore` (`.venv`, `.mypy_cache`, `.pytest_cache`, `.ruff_cache`, `dist/`, `.worktrees/`, `__pycache__`)
- Stale migration output directories at repo root (`migration_output_gpt4o_to_qwen*`)
- Screenshots at the repo root that should live under `docs/images/` or be deleted
- Duplicate / parallel code paths (investigate `server/` vs `server_stitch/` specifically — determine which is live, recommend deletion of the other)
- Dead code, unreferenced modules, abandoned experiments
- Dependencies in `pyproject.toml` not actually imported
- Optional extras that no longer earn their keep
- Outdated docs or spec files
- Test files for removed features

**Research rules:**

- For each candidate, verify by grepping for references: `grep -rn "<module_or_file>" src/ tests/ docs/ pyproject.toml`
- Note the grep results in the report — they're evidence for the recommendation
- For ambiguous items where grep shows no references but the code *looks* load-bearing, mark `NEEDS_HUMAN_REVIEW` with a one-line reason
- Estimate size/impact where possible (disk savings, dependency reduction, line count removed)

For each item in the report, include:
- Path
- Size (if relevant)
- One-line justification
- Recommended action: `SAFE_DELETE`, `MOVE_TO` (with destination), `GITIGNORE_ONLY`, `REFACTOR`, or `NEEDS_HUMAN_REVIEW`
- Grep evidence (reference count, where referenced)
- Risk level (low/med/high)

Write findings to `docs/reviews/2026-04-06-product-review.md` under Pass 1 as you go.

## Guardrails (do not violate)

- **Plan only.** No source edits, no test changes, no file deletions outside `docs/reviews/`. If you find a bug, note it in the report with a `BROKEN` tag — do not fix it.
- **No test runs.** Reading is enough. Any claim about behavior should come from reading source, not from running anything.
- **Branch lock.** Never leave `review/product-2026-04-06`. No push, merge, rebase, or checkout of another branch.
- **Scope lock.** Write only to `docs/reviews/`. Do not touch `docs/reviews/2026-04-06-testing/` — that directory belongs to the parallel testing review instance.
- **Grep for evidence.** Any claim that something is dead code, unreferenced, or duplicated needs a grep result in the report as evidence.
- **Ambiguity default.** When in doubt, mark `NEEDS_HUMAN_REVIEW` with a one-line question and move on.
- **No scope creep.** Do not invent features, expand the review lenses, or drift into testing-review territory. Stay inside the five passes.
- **Compact at phase boundaries only.** Never mid-pass. If you must, commit a `WIP` marker first.
- **Worklog before compact.** Always update the worklog Current Status and write a Phase Summary before running `/compact`.
- **Post-compact re-orientation is mandatory.** Read active prompt, read worklog, check git log, check git status, then resume.
- **Tone.** Direct, plainspoken, no hedging filler. Avoid the words *delve*, *leverage*, *unlock*, *synergy*, *holistic*, *robust*, *seamless*, *empower*. Say what you mean.
- **Length discipline.** As long as needed and no longer. Cut anything restating instead of saying something new.
